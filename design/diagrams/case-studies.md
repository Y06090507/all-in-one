# 实战读图案例

用真实场景练读图。

## 案例 1：GitHub 登录流程

```
┌──────┐     ┌──────────┐     ┌──────────┐     ┌────────┐
│ 用户  │     │ GitHub   │     │  OAuth   │     │ 数据库  │
│      │     │  Client  │     │  Server  │     │        │
└──┬───┘     └────┬─────┘     └────┬─────┘     └───┬────┘
   │  点击登录     │                │               │
   │─────────────>│                │               │
   │              │ 重定向授权页     │               │
   │<─────────────│                │               │
   │  输入账号密码  │                │               │
   │──────────────────────────────>│               │
   │              │                │ 验证用户       │
   │              │                │──────────────>│
   │              │                │  用户数据      │
   │              │                │<──────────────│
   │              │ 返回 code      │               │
   │<──────────────────────────────│               │
   │  携带 code    │                │               │
   │─────────────>│                │               │
   │              │ 用 code 换 token               │
   │              │───────────────>│               │
   │              │  access_token │               │
   │              │<───────────────│               │
   │  登录成功     │                │               │
   │<─────────────│                │               │
```

**读图**:
1. 参与者 4 个：用户、Client、OAuth Server、数据库
2. 用户只和 Client 交互，不直接碰 Server
3. 关键步骤：code 换 token（安全设计）
4. 时序：从上到下，6 个大步骤

## 案例 2：Redis 缓存穿透防护

```
         ┌──────────────┐
         │   请求进入    │
         └──────┬───────┘
                │
         ┌──────▼───────┐
         │ 查 Redis     │
         │ 有数据？      │
         └──────┬───────┘
           有  /   \  无
              /     \
     ┌───────┘       └───────┐
     │                       │
┌────▼────┐           ┌──────▼──────┐
│ 返回数据 │           │ 查 MySQL    │
└─────────┘           │ 有数据？     │
                      └──────┬──────┘
                         有 /   \ 无
                           /     \
                  ┌───────┘       └───────┐
                  │                       │
          ┌───────▼──────┐        ┌───────▼──────┐
          │ 写回 Redis    │        │ 返回空/默认值 │
          │ 返回数据       │        │ 防止缓存穿透  │
          └───────────────┘        └──────────────┘
```

**读图**:
1. 两层判断（菱形）
2. 缓存命中（有）→ 直接返回，快速路径
3. 缓存未命中（无）→ 查数据库 → 写回缓存
4. 数据库也没有 → 返回空值防止穿透（关键设计）

## 案例 3：微服务架构图

```
Internet
    │
┌───▼────────────────────────────────┐
│           API Gateway              │
│    (认证 / 限流 / 路由 / 日志)      │
└───┬──────┬──────┬──────┬──────────┘
    │      │      │      │
    ▼      ▼      ▼      ▼
┌──────┐┌──────┐┌──────┐┌──────────┐
│ 用户  ││ 订单  ││ 商品  ││  通知     │       ← 业务服务
│ 服务  ││ 服务  ││ 服务  ││  服务     │
└──┬───┘└──┬───┘└──┬───┘└───┬──────┘
   │       │       │        │
   ▼       ▼       ▼        ▼
┌──────┐┌──────┐┌──────┐┌──────────┐
│MySQL ││MySQL ││  ES  ││ RabbitMQ │       ← 数据层
└──────┘└──────┘└──────┘└──────────┘
                      │
              ┌───────▼───────┐
              │     Redis     │           ← 缓存层
              └───────────────┘
```

**读图**:
1. 入口：Internet → API Gateway (所有流量经过)
2. 水平拆分：4 个独立服务、各自数据库
3. ES 给商品服务做搜索
4. RabbitMQ 是通知服务的消息通道
5. Redis 是共享缓存、跨服务访问

## 练习：你自己读

看下面这张图，回答三个问题：

```
┌──────────┐     ┌──────────┐
│  User    │ 1  *│  Order   │
│──────────│─────│──────────│
│ id (PK)  │     │ id (PK)  │
│ name     │     │ user_id  │
│ email    │     │ total    │
└──────────┘     │ status   │
                 └────┬─────┘
                      │ 1
                      │
                 ┌────▼─────┐
                 │OrderItem │
                 │──────────│
                 │order_id  │
                 │product_id│
                 │quantity  │
                 └──────────┘
```

1. 这是什么图？(提示: 两种)
2. User 和 Order 什么关系？
3. Order 和 OrderItem 之间 `1 *` 啥意思？

<details>
<summary>答案</summary>

1. **ER 图** (数据库) + **类图** (实体关系)
2. User 1 → * Order：**一对多**，一个用户多个订单
3. Order 1 → * OrderItem：一个订单包含多个商品项

读法：先看方框(实体)、再看连线数字(关系)、PK 靠前放。
</details>
