# UML 统一建模语言

软件设计的「工程图纸」，用于在编码前把系统结构说清楚。

## 为什么学 UML？

- 代码写之前先画图 → 少走弯路
- 团队沟通的共同语言
- 文档即图纸，接手项目不懵逼

## UML 图分类

### 结构图 (系统长什么样)

| 图名 | 用途 | 关键词 |
|------|------|--------|
| **类图** Class | 类与类的关系 | 继承/实现/聚合/依赖 |
| **对象图** Object | 某时刻的对象快照 | 实例/值 |
| **组件图** Component | 系统模块划分 | 接口/依赖 |
| **部署图** Deployment | 服务器/节点拓扑 | 节点/协议 |
| **包图** Package | 代码分包结构 | 依赖/导入 |

### 行为图 (系统怎么运行)

| 图名 | 用途 | 关键词 |
|------|------|--------|
| **时序图** Sequence | 对象间消息传递顺序 | 生命线/激活/消息 |
| **用例图** Use Case | 用户能做什么 | 参与者/用例/包含/扩展 |
| **活动图** Activity | 业务流程/算法流程 | 分支/汇合/泳道 |
| **状态图** State | 对象生命周期 | 状态/转换/事件 |

## 常用程度排名

```
🥇 类图      — 90% 的场景都在画
🥈 时序图    — 接口调用流程
🥉 用例图    — 需求分析
4   活动图    — 复杂业务流程
5   状态图    — 订单/工单等状态机
```

## 画图原则

1. **先简后繁** — 别一上来就画全，先画核心
2. **一张图一个视角** — 别把结构+行为画一起
3. **标注清楚** — 关系线要写含义 (extends/implements/uses)
4. **代码能对上** — 图是代码的地图，不是装饰
